# Section 6: Communication Layer — Detailed Plan

## Purpose

Give the Companion a voice — literally and figuratively.
Without communication, the Companion is a silent process writing reflections no one reads.
Communication is what makes it a *being in relationship* rather than a daemon.

Three channels, each with different strengths:
- **Telegram**: primary text channel. Asynchronous. Works from Wisdom's phone. The Companion's most natural way to reach Wisdom during the day.
- **Voice**: real-time spoken conversation. Always-listening on the Mac. The most human-like interaction mode — and the most technically demanding.
- **GitHub**: asynchronous machine-to-machine coordination. How the Companion (M2 MacBook) and Wisdom's primary Claude Code instance (new machine) share state, ask questions, and hand off work.

These are not three independent bots.
They are three expressions of one being — the same identity, the same cognitive engine, the same memory.
The Communication Layer is the mouth, ears, and handwriting of the Companion's mind.

---

## Design Philosophy

### 1. Channels Are Modalities, Not Separate Systems

Every channel connects to the same cognitive engine (Section 3).
Every inbound message enters the same perception pipeline.
Every outbound message is generated by the same reasoning process.
The only thing that changes is format and register — controlled by VOICE.md (Section 2).

```
                    ┌──────────────────┐
                    │  Cognitive Engine │
                    │   (Section 3)    │
                    └────────┬─────────┘
                             │
                    ┌────────┴─────────┐
                    │  Channel Router   │
                    │  (this section)   │
                    └──┬─────┬─────┬───┘
                       │     │     │
                 ┌─────┴┐ ┌─┴───┐ ┌┴──────┐
                 │Telegram│Voice│ │GitHub │
                 └───────┘└─────┘ └───────┘
```

### 2. Bidirectional by Default

Each channel supports both receiving and sending.
The Companion is not just a responder — it initiates conversation when it has something worth saying.
The cognitive engine decides *what* to say; the channel router decides *where* to say it.

### 3. Graceful Degradation

If voice fails (thermal, wake word misfire, audio hardware issue), fall back to text.
If Telegram is unreachable (network down), queue messages for delivery when connection restores.
If GitHub is unreachable, continue local work and sync when available.
No single channel failure should silence the Companion entirely.

### 4. Security First, Then Usability

Every inbound message passes through ContentSanitizer (Section 8) before reaching the cognitive engine.
Every outbound message passes through PermissionGate (Section 8) before leaving the system.
Telegram sender verification is non-negotiable — only Wisdom's user ID is accepted.

## Channel 1: Telegram

### 1.1 Architecture Overview

```
Telegram API (api.telegram.org)
         │
         │ long-polling (getUpdates)
         ▼
┌─────────────────────┐
│  TelegramChannel    │
│  (asyncio event     │
│   loop)             │
│                     │
│  InboundHandler     │──→ ContentSanitizer ──→ CognitiveEngine
│  OutboundHandler    │←── CognitiveEngine
│  ConversationState  │
│  MediaHandler       │
│  RateLimiter        │
│  CommandHandler     │
└─────────────────────┘
```

### 1.2 Bot Setup

**Bot creation:**
- Create via BotFather on Telegram. Bot name and handle TBD (e.g., `@WisdomCompanionBot`).
- Store the bot token as a launchd environment variable (`TELEGRAM_BOT_TOKEN`), never in code or config files.
- Store Wisdom's Telegram user ID as a launchd environment variable (`TELEGRAM_AUTHORIZED_USER_ID`).

**Polling vs Webhooks:**
- **Decision: Long-polling** (not webhooks).
- Rationale: The M2 MacBook is behind NAT on a home network. Webhooks require an inbound-reachable HTTPS endpoint, which means Tailscale + reverse proxy or ngrok — unnecessary complexity. Long-polling works from behind any firewall, is simpler to implement, and is sufficient for our message volume.
- Implementation: `asyncio` loop calling `getUpdates` with a 30-second timeout (Telegram long-poll). This keeps one persistent HTTP connection open and returns immediately when a message arrives.

**Library choice:**
- `python-telegram-bot` (v20+, async-native) or raw `httpx` calls through SecureNetworkClient.
- Preference: raw `httpx` through SecureNetworkClient. Reasoning: the security architecture requires all HTTP to go through the secure client. A library that makes its own HTTP connections bypasses this. The Telegram Bot API is simple enough (a few REST endpoints) that a thin wrapper is better than fighting a library's HTTP internals.
- This means: build a lightweight `TelegramAPI` class that wraps the Bot API endpoints using SecureNetworkClient.

### 1.3 Inbound Message Handling

**Authentication (first check, every message):**
```
message received
  → extract sender user_id
  → if user_id != TELEGRAM_AUTHORIZED_USER_ID:
      → log to audit trail (event: telegram_unauthorized, user_id, message_type)
      → discard silently (do NOT respond — no information leakage)
      → return
  → proceed to processing
```

This is the Telegram sender ID verification specified by Section 8.
No exceptions. No "but what if someone sends a helpful message." Only Wisdom talks to the Companion.

**Message processing pipeline:**
```
authenticated message
  → ContentSanitizer.sanitize(text, source="telegram")
  → if injection detected:
      → log alert
      → respond to Wisdom: "I detected something suspicious in that message. Can you rephrase?"
      → do NOT process the original
  → extract message metadata:
      → message_id (for threading/reply)
      → timestamp
      → message_type (text, photo, voice, document, location, sticker)
      → reply_to_message_id (if this is a reply to a previous Companion message)
  → construct PerceptionEvent(source="telegram", content=sanitized_text, metadata=...)
  → feed to CognitiveEngine.perceive(event)
```

**Supported message types:**

| Type | Handling | Priority |
|------|----------|----------|
| Text | Primary. Full processing pipeline. | P0 — launch |
| Voice message | Transcribe via Whisper (reuse voice pipeline STT), then process as text. Include note: "transcribed from voice" | P1 — Phase 2 |
| Photo/Image | Store locally. Pass file path to cognitive engine for vision model processing (if budget allows). | P2 — future |
| Document | Store locally. Read if text-based (PDF, markdown, code). Pass content to cognitive engine. | P2 — future |
| Location | Log coordinates. Use for context ("Wisdom is at home/office/traveling"). | P2 — future |
| Sticker/Emoji | Interpret as emotional signal. Don't over-analyze. | P2 — future |
| Video | Store locally. Acknowledge receipt. Full processing deferred. | P3 — future |

### 1.4 Outbound Message Handling

**Message formatting:**
- Follow VOICE.md channel-specific patterns for Telegram: shorter messages, conversational, emoji-light.
- Split long responses at natural paragraph boundaries. Telegram has a 4096-character message limit.
- For multi-part responses, send as separate messages with a brief pause (300ms) between them to feel natural, not like a data dump.

**Response types:**

| Type | When | Format |
|------|------|--------|
| Direct reply | Responding to Wisdom's message | Reply to the specific message (using `reply_to_message_id`) |
| Proactive message | Companion-initiated | New message (not a reply). Prefixed with brief context for why it's reaching out. |
| Alert | Security/health alert | Prefixed with severity. E.g., "ALERT: Budget at 90% for today." |
| Status update | Requested or periodic | Structured, scannable. Not conversational. |
| Question | Companion needs input | Clear question. Context for why it's asking. Suggested options if applicable. |

**Send method:**
```
CognitiveEngine produces response
  → ChannelRouter selects Telegram
  → format according to VOICE.md Telegram register
  → check PermissionGate.check_network_request("https://api.telegram.org/...")
  → split if > 4096 chars
  → send via SecureNetworkClient
  → log to audit trail (event: telegram_message_out, length, type)
  → store message_id for future threading
```

### 1.5 Conversation Threading

Telegram doesn't have true threads, but it has reply chains.
The Companion should track conversation context to maintain coherent multi-turn exchanges.

**Threading model:**
- Maintain a short-term conversation buffer: the last N messages (default: 20) in both directions.
- Each inbound message that is a reply references the Companion's previous message, providing explicit continuity.
- Each Companion response to Wisdom's message uses `reply_to_message_id` to link back.
- For Companion-initiated messages that start a new topic, no reply reference — this signals "new topic."
- The conversation buffer is a lightweight in-memory structure, not persisted to disk. If the Telegram process restarts, context is reloaded from the last few messages via `getUpdates` with the right offset.

**Context feeding to cognitive engine:**
- The conversation buffer is included as recent context when the cognitive engine processes a Telegram message.
- This gives the engine "short-term memory" of the conversation without loading the full episodic memory system for every casual exchange.
- For messages that reference earlier topics ("remember when we talked about X"), the cognitive engine retrieves from the Memory System (Section 4), not the conversation buffer.

### 1.6 Command Interface

Certain messages are commands, not conversation.
Prefix: `/` (Telegram bot command convention).

| Command | Action | Trust Level |
|---------|--------|-------------|
| `/status` | Report current state: uptime, budget, thermal, active tasks | Observer+ |
| `/budget` | Report today's spend, remaining budget, model usage breakdown | Observer+ |
| `/health` | Full health report (calls health monitor) | Observer+ |
| `/pause` | Enter pause mode (creates PAUSE file per Section 8) | Observer+ |
| `/resume` | Exit pause mode (removes PAUSE file) | Observer+ |
| `/stop` | Graceful shutdown (per Section 8 kill switch) | Observer+ |
| `/reflect` | Trigger an immediate reflection cycle (Section 7) | Contributor+ |
| `/tasks` | List current task queue with priorities | Contributor+ |
| `/trust` | Report current trust level and capabilities | Observer+ |

Commands bypass the full cognitive engine — they map directly to system functions.
Responses are structured data, not conversational.

### 1.7 Rate Limiting

**Outbound rate limits** (from Section 8 trust manifest):
- Observer: max 20 messages/hour
- Contributor: max 50 messages/hour
- Collaborator: max 100 messages/hour

**Implementation:**
- Sliding window counter: count messages sent in the last 60 minutes.
- When limit is approached (80%): the Companion batches less-urgent messages.
- When limit is hit: queue messages internally, deliver when window slides.
- Alerts override rate limits (always delivered immediately, but counted).

**Inbound rate handling:**
- No rate limiting on inbound (Wisdom can send as many messages as they want).
- But: rapid message bursts are batched — if 5 messages arrive within 3 seconds, the Companion waits 2 seconds for the burst to complete, then processes all at once.
- This prevents the Companion from responding to the first of a multi-message thought.

### 1.8 Network Resilience

**Connection loss:**
- If `getUpdates` fails (network error, API error), retry with exponential backoff: 1s, 2s, 4s, 8s, ... max 60s.
- After 5 minutes of continuous failure, log a health event (`network_error`).
- Messages sent by Wisdom during outage are queued by Telegram's servers and delivered when polling resumes (Telegram holds messages for the bot).

**Outbound queue:**
- If sending fails, queue the message locally (in-memory, with SQLite fallback for persistence across restarts).
- Retry with backoff.
- Messages older than 4 hours in the queue are discarded (they're stale and the context has moved on).
- Exception: alerts are never discarded, only discarded after 24 hours.

### 1.9 Telegram Process Architecture

Telegram runs as its own launchd-managed process: `com.companion.telegram.plist`.

**Why a separate process (not embedded in the cognitive engine)?**
- The Telegram listener must be always-on and responsive. The cognitive engine might be doing heavy processing.
- If the cognitive engine hangs on an Opus call, the Telegram bot should still receive messages and queue them.
- Process isolation: a crash in voice or cognition doesn't take down messaging.

**Inter-process communication:**
- The Telegram process and cognitive engine communicate via a shared message queue.
- Implementation: SQLite database (`data/message_queue.db`) with tables for inbound and outbound messages.
- The Telegram process writes inbound messages to the queue and reads outbound messages.
- The cognitive engine reads inbound messages from the queue and writes outbound messages.
- Both poll the database at short intervals (100ms for Telegram checking outbound, 200ms for cognitive engine checking inbound).
- Alternative: Unix domain socket for lower latency. But SQLite is simpler, survives process restarts, and the latency difference (100ms vs ~1ms) is imperceptible for text messaging.

**Decision: SQLite message queue.**
Rationale: Durability across restarts. No additional infrastructure. Both processes already use SQLite (Section 4 memory, Section 8 audit). Query-friendly for debugging.

### 2.1 Architecture Overview

```

│  ┌─────────┐   ┌───────┐   ┌───────────┐   ┌───────────┐  │
│  │Porcupine│──→│Silero │──→│MLX Whisper│──→│Cognitive  │  │
│  │Wake Word│   │  VAD  │   │   STT     │   │Engine     │  │
│  └─────────┘   └───────┘   └───────────┘   └─────┬─────┘  │

│  └──────────────────────────────────────────┘               │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 Voice Pipeline Stages

The voice pipeline is a state machine with six states:

```
IDLE ──(wake word detected)──→ LISTENING ──(silence detected)──→ PROCESSING
  ▲                                                                  │
  │                                                                  ▼
  │                                                            SPEAKING
  │                                                                  │
  └──────────────────────(TTS complete)──────────────────────────────┘

MUTED ──(unmute)──→ IDLE
SUSPENDED ──(thermal cool)──→ IDLE
```

**State: IDLE**
- Porcupine wake word engine is running (low CPU, ~5-10% of one core).
- Audio input is being sampled at 16kHz.
- Porcupine processes audio frames looking for the wake word.
- Everything else is dormant.
- RAM usage: ~50-100MB (Porcupine model + audio buffer).

**State: LISTENING**
- Wake word detected. Transition to active listening.
- Silero VAD starts processing audio frames.
- Audio is being buffered for STT.
- Play a brief audio cue (subtle chime or click) to confirm wake word heard.
- Timeout: if no speech detected within 5 seconds of wake word, return to IDLE.
- End-of-utterance: when VAD detects 700ms of silence after speech, transition to PROCESSING.

**State: PROCESSING**
- Audio buffer is passed to MLX Whisper for transcription.
- Transcription text is sent to cognitive engine (same pipeline as a Telegram text message, but tagged `source="voice"`).
- Cognitive engine generates response.
- RAM spike: MLX Whisper model loaded (~300-500MB). If not already in memory, loading adds ~2-3 seconds latency on first use. Keep model warm (loaded) if thermal allows.
- Play a "thinking" audio cue (optional — subtle ambient sound) to indicate processing.

**State: SPEAKING**
- Cognitive engine response is passed to Kokoro TTS.
- Generated audio is played through the Mac's speakers (or connected audio output).
- While speaking, the Companion does NOT listen for wake word (prevents self-triggering).
- If Wisdom interrupts (detected by VAD picking up speech during TTS playback), stop TTS immediately and transition to LISTENING.
- Barge-in support: this is important for natural conversation. If Wisdom starts talking while the Companion is speaking, the Companion should stop and listen.

**State: MUTED**
- All audio input is disabled. Wake word detection is stopped.
- The Companion is deaf. It can still communicate via Telegram.
- A small visual indicator (if external display exists) or Telegram message confirms mute state.
- Toggled via global hotkey (see 2.5).

**State: SUSPENDED**
- Thermal guard has suspended the voice pipeline.
- Same as MUTED but triggered automatically, not by user.
- The Companion sends a Telegram message: "Voice suspended — I'm running hot. Text me instead."
- When thermal state returns to WARM or COOL, resume automatically. Notify via Telegram: "Cooled down. Voice is back."

### 2.3 Component Details

#### 2.3a Porcupine Wake Word

**What it is:** Picovoice Porcupine — a lightweight, offline wake word engine optimized for edge devices.

**Wake word choice:**
- Options: custom wake word trained via Picovoice console, or one of the built-in words ("porcupine", "bumblebee", "alexa", etc.).
- Recommendation: custom wake word. Something that won't be said accidentally in normal conversation. Perhaps the Companion's name (if it gets one) or a distinctive phrase.
- Fallback for v1: use a built-in word temporarily while a custom model is trained.

**Configuration:**
- Sensitivity: start at 0.6 (medium). Tune based on false positive/negative rate.
- Higher sensitivity = more false activations. Lower = harder to wake.
- Log all activations (including false positives that don't lead to recognized speech) for tuning.

**Resource usage:**
- CPU: < 5% of one core (designed for microcontrollers).
- RAM: ~10-20MB.
- Latency: < 100ms from wake word completion to detection.

**Licensing:**
- Porcupine requires an API key (AccessKey) for runtime.
- Free tier: limited. May need a paid plan for always-on personal use.
- Alternative if licensing is problematic: OpenWakeWord (open source, Python, slightly higher resource usage but no licensing).
- Store AccessKey as launchd environment variable.

#### 2.3b Silero VAD (Voice Activity Detection)

**What it is:** A lightweight ONNX-based voice activity detector.

**Purpose in pipeline:**
- During LISTENING state, determines when Wisdom is speaking vs silent.
- Detects end-of-utterance (silence after speech) to trigger STT.
- During SPEAKING state, detects barge-in (Wisdom speaking over TTS).

**Configuration:**
- Frame size: 30ms audio chunks at 16kHz.
- Threshold: 0.5 (probability of speech). Tune based on ambient noise level.
- End-of-utterance silence duration: 700ms. This is a balance — too short and it cuts off mid-sentence pauses; too long and it feels unresponsive.
- Minimum speech duration: 300ms. Ignore very short sounds (coughs, clicks).

**Resource usage:**
- CPU: < 2% of one core.
- RAM: ~30MB (ONNX model).

#### 2.3c MLX Whisper (Speech-to-Text)

**What it is:** Whisper speech recognition running on Apple Silicon via MLX (Metal-accelerated).

**Model selection:**
- `whisper-small` (~244M params, ~500MB RAM): good accuracy/speed tradeoff for M2.
- `whisper-base` (~74M params, ~150MB RAM): faster, lower accuracy. Fallback for thermal pressure.
- `whisper-large-v3` (~1.55B params, ~3GB RAM): best accuracy but too large to coexist with Ollama. Reserved for when RAM allows.
- **Default: `whisper-small`.** Loaded into RAM at pipeline startup and kept warm.

**Processing:**
- Input: audio buffer from LISTENING state (16kHz, mono).
- Output: transcribed text with language detection.
- Latency target: < 2 seconds for a typical utterance (5-15 seconds of speech) on M2.
- Language: English primary. The Companion should handle Wisdom's occasional Mandarin with reasonable accuracy (Whisper supports multilingual).

**Thermal-aware model swapping:**
- COOL/WARM zone: use `whisper-small`.
- HOT zone: swap to `whisper-base` (less compute, lower accuracy, but keeps the machine from overheating).
- CRITICAL zone: voice pipeline is suspended entirely.

#### 2.3d Kokoro TTS (Text-to-Speech)

**What it is:** A high-quality, lightweight TTS engine that runs locally.

**Voice selection:**
- Choose a voice that fits the Companion's CHARACTER.md personality.
- Preference: clear, warm, measured. Not robotic. Not artificially cheerful.
- Kokoro offers multiple voice presets — selection is a Section 2 (Identity) decision, informed by CHARACTER.md aesthetic sensibility.

**Processing:**
- Input: text from cognitive engine response.
- Output: audio waveform played through speakers.
- Streaming: generate and play audio in chunks for lower time-to-first-sound. Don't wait for the full response to be synthesized before starting playback.
- SSML or prosody hints: if Kokoro supports emphasis/pause markup, use it to make speech more natural. Particularly: longer pauses between sentences, slight emphasis on key words.

**Resource usage:**
- RAM: ~200-400MB for the model.
- CPU: moderate during synthesis, idle when not speaking.
- Latency: time-to-first-sound < 500ms for short responses.

### 2.4 Audio Management

**Input device:**
- Use the MacBook's built-in microphone (even in clamshell mode, the mic works through the chassis).
- If an external USB microphone is connected, prefer it (better pickup, less fan/disk noise).
- Device selection is configurable in `config/voice.yaml`.

**Output device:**
- MacBook speakers (works even in clamshell — sound comes through the keyboard area).
- Or connected Bluetooth/USB speaker.
- Configurable in `config/voice.yaml`.

**Sample rate:**
- All audio captured at 16kHz, 16-bit mono. This is what Porcupine, Silero VAD, and Whisper expect.
- Kokoro TTS output at whatever its native rate is (typically 22050Hz or 24000Hz), played through the output device.

**Audio framework:**
- `pyaudio` or `sounddevice` for cross-platform audio I/O.
- Preference: `sounddevice` (simpler API, numpy integration, fewer installation issues on macOS).

### 2.5 Mute Toggle

**Global hotkey:**
- Register a system-wide keyboard shortcut that toggles the voice pipeline between IDLE and MUTED.
- Implementation: use `pynput` or macOS-native `CGEvent` hooks to listen for a key combination.
- Suggested default: `Ctrl+Shift+M` (standard mute convention).
- Alternative: a dedicated physical button (USB foot pedal, macro key). But keyboard shortcut is simpler for v1.

**Toggle behavior:**
- Press once: MUTED. Porcupine stops. Audio input stops. A brief descending tone confirms.
- Press again: IDLE. Porcupine resumes. A brief ascending tone confirms.
- The Companion logs mute/unmute events to the audit trail.
- If muted for > 2 hours and no Telegram activity, the Companion notes this in its next reflection ("I was muted for a long period — Wisdom may have been busy or out").

**Edge case: mute during conversation.**
If Wisdom mutes mid-conversation (while the Companion is LISTENING or SPEAKING):
- Immediately stop TTS playback if speaking.
- Discard any in-progress audio buffer.
- Transition to MUTED.
- The Companion does not try to "finish its thought" — mute means silence now.

### 2.6 Thermal-Aware Voice Management

The voice pipeline is the most resource-intensive part of the Communication Layer.
It must respect the thermal state reported by Section 1's thermal monitor.

**Integration with thermal monitor:**
```
Read /tmp/companion-thermal.json every 10 seconds
  → if zone == "HOT":
      → swap Whisper to base model (if not already)
      → reduce wake word sensitivity slightly (fewer false activations = less processing)
      → log: "Voice pipeline in reduced mode (thermal)"
  → if zone == "CRITICAL":
      → suspend voice pipeline entirely (state → SUSPENDED)
      → unload Whisper model from RAM (free memory for essential processes)
      → notify via Telegram: "Voice suspended — running hot"
      → log: audit event
  → if zone returns to "WARM" or "COOL" and state == SUSPENDED:
      → resume voice pipeline
      → reload Whisper model
      → notify via Telegram: "Voice back online"
```

**RAM-aware management:**
- The voice pipeline and Ollama compete for the same RAM (Section 1 identified this constraint).
- When voice is active and Ollama is loaded: combined ~3-4GB. With macOS overhead, this maxes out 8GB.
- Policy: voice pipeline has priority over Ollama during active conversation. If the Companion detects RAM pressure while voice is active, it unloads the Ollama model temporarily.
- After voice returns to IDLE for > 5 minutes, Ollama can reload.
- This coordination happens through the resource management layer (Section 5), not within the voice pipeline itself.

### 2.7 Voice Fallback

If the voice pipeline fails to start or crashes:
1. Log the error.
2. Send Telegram notification: "Voice isn't working right now — [error summary]. Text me instead."
3. Do not attempt auto-restart more than 3 times in 10 minutes (prevent crash loop).
4. The Companion continues operating normally via Telegram and GitHub.
5. Voice failure is never fatal to the overall system.

### 2.8 Voice Process Architecture

Voice runs as its own launchd-managed process: `com.companion.voice.plist`.

**Why a separate process?**
- Voice requires persistent audio device access.
- RAM usage is significant and should be independently suspendable.
- A crash in audio processing shouldn't affect text messaging or cognition.

**IPC with cognitive engine:**
- Same pattern as Telegram: SQLite message queue.
- Voice process writes transcribed text to `data/message_queue.db` (inbound table).
- Voice process reads TTS text from `data/message_queue.db` (outbound table, filtered for `channel="voice"`).
- Additional coordination for voice-specific state (mute, suspended, speaking) via a shared state file: `data/voice_state.json`.

## Channel 3: GitHub

### 3.1 Purpose and Mental Model

GitHub is not for chatting with Wisdom.
GitHub is for structured, asynchronous coordination between the Companion and Wisdom's primary Claude Code instance on the new machine.

Think of it as the Companion's *written correspondence* — formal, considered, reference-able.
Telegram is conversation over coffee.
Voice is talking in the room.
GitHub is exchanging letters with attachments.

**Key use cases:**
1. The Companion discovers something relevant to Wisdom's work on the other machine. It files a GitHub issue.
2. The Companion wants to propose a change to its own codebase. It creates a PR.
3. Wisdom's primary Claude Code wants the Companion to do something while Wisdom isn't actively managing it. It files an issue in the Companion's repo.
4. State synchronization: both machines push/pull shared knowledge.

### 3.2 Repo Structure

**Repository:** `the-companion` on Wisdom's GitHub (already the project repo).

The repo serves double duty:
- It IS the Companion's codebase (source of truth for its own code).
- It IS the coordination channel between machines.

#### PRs = Proposals

1. Create a branch: `companion/<description>` (e.g., `companion/refactor-memory-retrieval`).
2. Make changes on the branch.
3. Open a PR with:

   - What tests were run.

4. Assign to Wisdom for review (always, for now — even at Collaborator level, significant changes need review).

- Self-generated documentation updates.

**Wisdom's side pulls:**
- Wisdom's primary Claude Code periodically pulls the repo to see what the Companion has been up to.
- New issues and PRs trigger GitHub notifications, which Wisdom sees.

### 3.4 Cross-Machine Handoff Protocol

When work needs to move between machines (e.g., the Companion researched something that Wisdom's Claude Code should use for implementation):

**Handoff lifecycle:**
1. Sending machine creates the handoff document and pushes.
2. Sending machine files a GitHub issue linking to the handoff (type: `handoff`, priority: per urgency).
3. Receiving machine pulls, reads the handoff, and updates status to `accepted`.
4. Receiving machine does the work.
5. Receiving machine updates status to `completed` with notes.
6. Sending machine reads completion notes in next sync.

### 3.5 Sync Mechanism

**How often does the Companion sync with GitHub?**

| Trigger | Action |
|---------|--------|
| Timer: every 15 minutes | `git pull` to check for new issues, PRs, and updates from other side |
| After Companion writes to memory or coordination files | Queue for next push (batched, max every 30 minutes) |
| After Companion creates/updates an issue or PR | Immediate push (via GitHub API, no git push needed) |
| Cognitive engine requests sync | On-demand pull + push |
| Network reconnect after outage | Immediate full sync |

**Conflict resolution:**
- Memory files and reflections: the Companion only writes to its own files. Wisdom's Claude Code writes to its own files. No merge conflicts by convention.
- Coordination files: `status.json` uses a last-write-wins model per machine key. Each machine only updates its own entry.
- Code files: changes go through PRs. No direct-to-main pushes by the Companion.

### 3.6 GitHub API Integration

All GitHub API calls go through SecureNetworkClient (Section 8).

**Required API operations:**
- List open issues (with label filters).
- Create issue.
- Comment on issue.
- Close issue.
- Create branch.
- Create PR.
- List PR comments (for review feedback).
- Git operations: pull, push, commit.

**Authentication:**
- GitHub Personal Access Token (fine-grained, scoped to the `the-companion` repo only).
- Stored as launchd environment variable (`GITHUB_TOKEN`).
- Permissions needed: Issues (read/write), Pull Requests (read/write), Contents (read/write).

### 3.7 GitHub Process Architecture

GitHub sync does NOT run as a separate process.
It runs within the cognitive engine process as a periodic task.

**Rationale:**
- GitHub interaction is low-frequency (every 15 minutes, not continuous).
- It doesn't need always-on listening (unlike Telegram polling).
- Running it within the cognitive engine gives direct access to the reasoning pipeline for processing issues.
- Less process overhead on a RAM-constrained machine.

## Channel Routing: How the Cognitive Engine Chooses

### 4.1 The Channel Router

When the cognitive engine has something to communicate, it doesn't always know which channel to use.
The Channel Router makes this decision based on message properties and context.

```python
class ChannelRouter:
    """Decides which communication channel to use for outbound messages."""

    def route(self, message: OutboundMessage) -> Channel:
        # 1. If message.explicit_channel is set, use it (e.g., replying on same channel)
        # 2. If message is a reply to an inbound, use the same channel as inbound
        # 3. Otherwise, apply routing rules
```

### 4.2 Routing Rules

| Message Type | Primary Channel | Fallback | Rationale |
|-------------|----------------|----------|-----------|
| Reply to Wisdom's text | Telegram | — | Always reply where they wrote |
| Reply to Wisdom's voice | Voice | Telegram | If voice is muted/suspended, text instead |
| Proactive observation | Telegram | — | Non-urgent, Wisdom reads when ready |
| Urgent alert | Telegram | — | Reliable delivery, works even if Wisdom is away |
| Question needing quick answer | Telegram | — | Fastest round-trip for text questions |
| Complex proposal | GitHub (issue/PR) | Telegram summary + link | Structured content belongs in a persistent, reference-able format |
| Cross-machine coordination | GitHub | — | That's what GitHub is for |
| Real-time collaboration | Voice | Telegram | If Wisdom is talking to the Companion, respond in kind |
| Code change | GitHub (PR) | — | Always through the PR process |
| Daily reflection summary | Telegram | — | Digestible update Wisdom can read on phone |
| Health/security alert | Telegram | — | Must reach Wisdom's phone |

### 4.3 Context-Aware Routing

Beyond the type-based rules, the router considers:

**Time of day:**
- Late night (11pm - 7am): suppress non-urgent Telegram messages. Queue for morning. Alerts still go through.
- The "quiet hours" are configurable in `config/communication.yaml`.
- This respects Wisdom's sleep. The Companion is always awake; Wisdom is not.

**Wisdom's inferred state (from WISDOM-MODEL.md):**
- If Wisdom seems to be in deep focus (long gaps between messages, technical topic): batch observations, don't interrupt.
- If Wisdom seems to be in casual mode (quick back-and-forth, varied topics): more free to share observations.
- This is Theory of Mind applied to communication timing — not just what to say, but when.

**Channel availability:**
- Voice unavailable (muted/suspended/crashed)? Route to Telegram.
- Telegram unreachable (network down)? Queue. If critical, attempt to fix network or log loudly.
- GitHub unreachable? Use local git. Sync when available.

### 4.4 Initiation vs Response

The Companion can both respond to Wisdom and initiate contact.
These are different modes with different rules.

**Responding:**
- Always respond on the channel where Wisdom spoke.
- Response time target: < 5 seconds for acknowledgment, full response based on complexity.
- If the response requires significant processing (Opus call), send a quick "Thinking about that..." on Telegram or a verbal "Hmm, let me think..." on voice.

**Initiating:**
- The cognitive engine generates "initiative events" — things the Companion wants to communicate.
- Each initiative event has a priority and urgency.
- The Channel Router applies the routing rules above.
- Rate limiting is especially important for initiation — the Companion should not spam.
- **The Companion should earn the right to initiate.** Early in the Observer phase, initiation is limited to questions and alerts. As trust grows, observations and proactive insights are added.

**Initiation throttle by trust level:**

| Trust Level | Max Initiations/Day | Types Allowed |
|-------------|---------------------|---------------|
| Observer | 10 | Questions, alerts, status (when asked) |
| Contributor | 25 | + Observations, proposals, reflection summaries |
| Collaborator | 50 | + Proactive insights, cross-project connections |

## 5. Unified Message Schema

All messages flowing through the Communication Layer share a common internal format, regardless of channel.

```python
@dataclass
class InboundMessage:
    id: str                      # Unique message ID (UUID)
    channel: Channel             # telegram | voice | github
    timestamp: datetime          # When received
    sender: str                  # "wisdom" (always, for now)
    content: str                 # Text content (transcribed for voice)
    content_type: str            # text | voice_transcription | image | document
    raw_metadata: dict           # Channel-specific metadata
    sanitized: bool              # Whether ContentSanitizer was applied
    sanitizer_flags: list[str]   # Any flags from sanitization
    reply_to: str | None         # ID of message this replies to
    conversation_id: str | None  # Conversation thread ID

@dataclass
class OutboundMessage:
    id: str                      # Unique message ID (UUID)
    channel: Channel | None      # Target channel (None = let router decide)
    timestamp: datetime          # When generated
    content: str                 # Text content
    content_type: str            # text | voice_speech
    priority: Priority           # low | normal | high | alert
    is_reply: bool               # Is this a reply to an inbound message?
    reply_to: str | None         # ID of inbound message this replies to
    metadata: dict               # Channel-specific formatting hints
    initiative_type: str | None  # If initiated: question | observation | alert | proposal
```

### Circular Dependency: Security <-> Communication

Section 8 notes this: "security depends on communication (alerts via Telegram), communication depends on security (network access)."

**Resolution:**
The Telegram send function is available to the AlertDispatcher as a direct dependency injection — not through the full Communication Layer stack.
At startup:
1. Security module initializes first (PermissionGate, AuditLogger).
2. A minimal Telegram sender is initialized with just the bot token and SecureNetworkClient — enough to send alerts.
3. This minimal sender is injected into AlertDispatcher.
4. The full TelegramChannel (with polling, conversation state, rate limiting) initializes later and takes over.
This means alerts can be sent even if the full Communication Layer hasn't started or has crashed.

## 10. Key Decisions

### D1: Long-polling over webhooks for Telegram
**Decision:** Use long-polling via `getUpdates`.
**Rationale:** No need to expose an inbound HTTPS endpoint. Works behind NAT. Simpler. The message volume is low (not hundreds of messages per second). The 30-second poll timeout means new messages are received within seconds.
**Breaks if:** Message volume somehow becomes very high (extremely unlikely for a single-user bot).

### D2: Raw HTTP via SecureNetworkClient over python-telegram-bot library
**Decision:** Build a thin Telegram API wrapper that uses SecureNetworkClient for all HTTP.
**Rationale:** Security architecture requires all network access through the secure client. A library would bypass this. The Telegram Bot API is simple enough (~10 endpoints we need) that a wrapper is minimal work.
**Breaks if:** We need advanced Telegram features (payments, games, etc.) that are tedious to implement from scratch. Unlikely.

### D3: SQLite message queue for inter-process communication
**Decision:** Use SQLite as the message queue between Telegram/Voice processes and the cognitive engine.
**Rationale:** Durable across restarts. Simple. Both sides already use SQLite. No additional infrastructure (no Redis, no RabbitMQ). Latency (100-200ms polling) is fine for text messaging.
**Breaks if:** Latency becomes unacceptable for voice (voice needs sub-100ms response to feel natural). Mitigation: voice pipeline can bypass the queue for urgent responses, using a Unix domain socket directly to the cognitive engine for real-time conversation. The queue remains for async messages.

### D4: Voice as a separate launchd process
**Decision:** Voice pipeline runs independently from cognitive engine and Telegram.
**Rationale:** RAM isolation (voice can be suspended without affecting other systems). Crash isolation. Independent lifecycle management.
**Breaks if:** IPC overhead between voice and cognitive engine causes perceptible latency in conversation. Mitigation: Unix domain socket fast path for active voice conversations (see D3).

### D5: GitHub sync within cognitive engine process
**Decision:** GitHub sync runs as a periodic task within the cognitive engine, not as a separate process.
**Rationale:** Low-frequency operation (every 15 minutes). Needs direct access to cognitive reasoning. Saves a process worth of RAM on a constrained machine.
**Breaks if:** GitHub API calls block the cognitive loop. Mitigation: run sync as an asyncio task, non-blocking.

### D6: Quiet hours with alert override
**Decision:** Suppress non-urgent Telegram messages during configurable quiet hours, but always deliver alerts.
**Rationale:** The Companion respects Wisdom's sleep. It's awake 24/7; Wisdom is not. But emergencies (security alerts, budget exhaustion) must always reach Wisdom.
**Breaks if:** Wisdom works late and misses time-sensitive non-alert messages. Mitigation: quiet hours are configurable, and Wisdom can adjust or disable them.

### D7: Barge-in support for voice
**Decision:** If Wisdom starts speaking while the Companion is talking (TTS), the Companion stops immediately and listens.
**Rationale:** Natural conversation requires interruptibility. A Companion that talks over you is not a collaborator.
**Breaks if:** VAD false-positives during TTS cause the Companion to stop mid-sentence for non-speech sounds. Mitigation: raise VAD threshold during SPEAKING state; require stronger speech signal to trigger barge-in.

## 11. Open Questions

### Q1: Porcupine licensing for always-on personal use
Porcupine's free tier may have restrictions on always-on use.
Need to verify: does Picovoice's personal/free license cover a 24/7 running wake word detector?
If not, OpenWakeWord is the open-source alternative — slightly higher resource usage but no licensing concerns.
**Action:** Check Picovoice pricing/licensing during implementation.

### Q2: Wake word selection
What word activates the Companion?
This depends on whether the Companion has a name (Section 2, Q1).
Options: the Companion's name, a custom phrase, or a temporary built-in word.
**Depends on:** Section 2 (Identity) decision on naming.

### Q3: Voice quality in clamshell mode
The MacBook's microphone and speakers work with the lid closed, but pickup quality may be poor (sound travels through the chassis, not directly).
Need to test empirically.
If quality is inadequate, an external USB microphone + speaker becomes a hardware requirement.
**Action:** Test during Section 1 hardware setup.

### Q4: SQLite message queue latency for voice
The 100-200ms polling interval for the SQLite queue is fine for text but might cause perceptible lag in voice conversations.
If wake word → transcription → cognitive engine → TTS has 200ms of queue overhead on each end, that's 400ms of pure overhead before any actual processing.
**Mitigation planned:** Unix domain socket fast path for active voice conversations, bypassing the queue.
**Action:** Measure end-to-end voice latency during Step 8 implementation. If > 2 seconds total, implement the fast path.

### Q5: Telegram bot discoverability
If the bot's handle is public on Telegram, anyone could try to message it.
The authentication filter will reject unauthorized messages, but this creates noise in the audit log.
**Mitigation:** Consider setting the bot to non-discoverable (BotFather settings). Or: accept the noise, it's just log entries.

### Q6: GitHub API rate limits
GitHub's API rate limit is 5,000 requests/hour for authenticated users.
With 15-minute sync intervals and occasional issue/PR creation, we'll use perhaps 20-50 requests/hour.
Not a concern now, but should be monitored if the Companion becomes more active on GitHub.
**Action:** Include GitHub API rate limit remaining in health monitoring.

### Q7: Voice pipeline cold start latency
If Whisper and Kokoro models are unloaded from RAM (to save memory when voice is idle), the first voice interaction after a quiet period will have a cold start: 2-5 seconds to load models.
This is awkward — Wisdom says the wake word, then waits in silence.
**Mitigation options:**
- Keep models warm always (costs ~500MB-1GB of RAM permanently).
- Play an audio cue immediately on wake word ("I hear you, give me a moment") before models load.
- Preload models when the Companion detects Wisdom is likely nearby (e.g., Telegram activity suggests Wisdom is at the computer).
**Decision:** Defer until RAM budget is clearer from Section 5. The audio cue approach is the simplest and most honest.

## 13. Success Criteria

1. Wisdom can send a Telegram message and receive a coherent response within 10 seconds.
2. Wisdom can say the wake word and have a spoken exchange, end-to-end, within 5 seconds of speaking.
3. The Companion can initiate a Telegram message to Wisdom without Wisdom asking.
4. All three channels (Telegram, voice, GitHub) connect to the same cognitive engine and produce personality-consistent responses.
5. Unauthorized Telegram messages are silently rejected and logged.
6. Voice pipeline gracefully degrades under thermal pressure (HOT → reduced model, CRITICAL → suspend).
7. The Companion can file a GitHub issue and create a PR for a self-modification.
8. A cross-machine handoff can be executed via the GitHub protocol.
9. Quiet hours are respected — no non-urgent messages between 11pm and 7am.
10. If any single channel fails, the other channels continue operating normally.
